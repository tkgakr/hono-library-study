# 11. まとめと発展

ここまでで、図書館 API を **4層アーキ + zod-openapi + Drizzle + neverthrow + bun:test** で一通り実装しきりました。最終章では全体を振り返り、本教材では**あえて省略した発展トピック**（実務で追加されることの多い要素）を整理します。

| 02 | 4層アーキ概観 | 依存の向き（内←外）、リクエストの流れ |

## 発展トピック（実務で追加されることの多い要素）

本教材は「4層と zod-openapi の本質」に集中するため、以下を省きました。実務ではこれらが**追加で乗る**ことがあります。各トピックは「本教材の構成からどう拡張するか」という観点でまとめます。

### 1. マルチテナント（テナントごとのスキーマ分離）

1つの DB を複数の組織・店舗・拠点などで共有しつつ、データを分離したいケースです。

| 観点 | 本教材 | マルチテナント化したとき |
| --- | --- | --- |
| テーブル接続 | `pgTable(...)` をそのまま | テナント ID からスキーマを決め、`pgSchema('tenant_<id>').table(tableConfig.name, tableConfig.columns)` のように接続する関数を用意 |
| リポジトリ | 引数なし | 各メソッドの第1引数で `tenantId` を受け取る |
| usecase | 引数なし | `tenantId` を受け取り repository へ渡す |
| ルーター | なし | パスパラメータやトークンから `tenantId` を取り出し、ミドルウェアで context にセット |

03章で `DatabaseTableConfig`（テーブル名 + カラム定義）を**型として分離**しておいたのは、まさにこの拡張点のためです。同じカラム定義を「デフォルトの `pgTable`」にも「テナントのスキーマ」にも渡せます。

### 2. 認証（JWT など）

| 観点 | 本教材 | 認証を加えたとき |
| --- | --- | --- |
| 認証 | なし | `serverIndex.ts` で全 API に検証ミドルウェア（例：Hono の `jwt()`）を適用 |
| エラーコード | なし | トークン未指定・形式不正・期限切れ等に対応する `W99xx` 帯を追加 |

Hono は `hono/jwt` を標準で備えており、`app.use(...)` でルート群にまとめて適用できます。検証失敗は本教材の `globalErrorHandler` / `setResponse` と同じ仕組みに乗せれば、レスポンス形式を統一できます。

### 3. 複合ステータスコード

本教材の `apiStatus.code` は `I0000` のような結果コードだけですが、実務では **どのリソースのどの操作で起きたか**を埋め込んだ複合コードにすることがあります（例 `03-01-I0000`）。

| 部品 | 役割 | 設定の考え方 |
| --- | --- | --- |
| `pathCode`（機能2桁） | どのリソースか | ルート集約でリソースごとのコードをミドルウェアで context にセット |
| `methodCode`（操作2桁） | どの操作か | 各ルートのミドルウェアで操作コードを context にセット |
| `resultCode` | 結果 | usecase が返す `OperationResult.code` |

`setResponse` で `${pathCode}-${methodCode}-${code}` のように連結します。context 変数（`pathCode`/`methodCode` など）を持ち回るのがその基盤です。クライアント側はコードを `-` で分割すれば、必要な部分（resultCode など）だけ取り出せます。

### 4. その他

| 項目 | 本教材 | 実務での発展 |
| --- | --- | --- |
| ログ | `console` | winston 等の構造化ログ基盤を `ILogger` 実装として注入 |
| 日付 | `new Date()` 直接 | 日付ライブラリを `IDateService` 実装として注入（テスト時に固定可能） |
| マイグレーション | drizzle-kit 標準 `migrate` | 複数スキーマ対応の独自マイグレーションスクリプト |
| 統合テスト | 省略 | 実 app + 環境モックで HTTP レベルの status / resultCode を検証 |

いずれも、本教材で作った**抽象（`ILogger` / `IDateService` / リポジトリ契約）に実装を差し替える**だけで足せます。これが4層アーキの恩恵です。

## 次の演習アイデア

- `loanCheckService` を作り、「貸出中の蔵書は再貸出できない」ルールを `.andThrough` で追加する
- `member` の4層を book を雛形に自力で写経する（定着の確認）
- `setPathCode`/`setMethodCode` 相当のミドルウェアを導入し、複合コードを再現してみる
- 統合テスト（実 app + 環境モック）を導入し、HTTP レベルで status と resultCode を検証する
- マルチテナント化：`DatabaseTableConfig` を使ってテナントごとのスキーマへ接続する関数を実装する

おつかれさまでした。これで `@hono/zod-openapi` を用いた4層バックエンドを「読める」だけでなく「同じ流儀で書ける」状態になっています。
