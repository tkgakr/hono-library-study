# 08. 動作確認 — book の縦1本を通す

04〜07章で book の4層が揃いました。この章では実際に動かして、**HTTP → interface → application → domain → infrastructure → DB** の縦1本が通ることを確認します。

## 1. 起動

```sh
docker compose up --build
```

`PORT 3000 で起動しました` が出て、SQL ログ（`DEBUG=true`）が流れ始めれば OK。マイグレーション未適用なら、別ターミナルで:

```sh
docker compose exec api bash
bun run db:migrate   # book テーブルを作成（05章で生成済みのSQLを適用）
```

## 2. Swagger UI で全体像を見る

ブラウザで `http://localhost:3000/api-docs` を開きます。`蔵書` タグの下に6本のエンドポイントが並びます。

| メソッド | パス | 操作 |
| --- | --- | --- |
| GET | `/books` | 一覧取得 |
| POST | `/books` | 作成 |
| GET | `/books/{id}` | 詳細取得 |
| PUT | `/books/{id}` | 更新 |
| DELETE | `/books/{id}` | 無効化（論理削除） |
| PUT | `/books/activate/{id}` | 復元 |

これらは `createRoute` の宣言から**自動生成**されたものです。`.openapi({ example })` で付けた例も表示されます。

## 3. curl でCRUDを一周する

### 作成

```sh
curl -s -X POST http://localhost:3000/books \
  -H 'Content-Type: application/json' \
  -d '{"title":"吾輩は猫である","author":"夏目漱石"}' | jq
```

```json
{
  "apiStatus": { "code": "I0000", "message": "正常に処理されました" }
}
```

### 一覧取得（作成した本が出る）

```sh
curl -s 'http://localhost:3000/books' | jq
```

```json
{
  "apiStatus": { "code": "I0000", "message": "正常に処理されました" },
  "data": {
    "value": [
      { "id": "...", "title": "吾輩は猫である", "author": "夏目漱石", "isActive": true }
    ],
    "total": 1
  }
}
```

`id` を控えておきます（以降 `$ID` と表記）。

### 詳細取得 / 更新 / 削除 / 復元

```sh
# 詳細
curl -s http://localhost:3000/books/$ID | jq

# 更新（author だけ変更）
curl -s -X PUT http://localhost:3000/books/$ID \
  -H 'Content-Type: application/json' -d '{"author":"夏目 漱石"}' | jq

# 無効化（論理削除：isActive=false になる）
curl -s -X DELETE http://localhost:3000/books/$ID | jq

# 復元
curl -s -X PUT http://localhost:3000/books/activate/$ID | jq
```

## 4. エラー系で「2段バリデーション」と「コード体系」を体感する

### 1段目: createRoute スキーマ違反（defaultHook → W9901）

`title` を欠いて作成:

```sh
curl -s -X POST http://localhost:3000/books \
  -H 'Content-Type: application/json' -d '{"author":"夏目漱石"}' | jq
```

```json
{ "apiStatus": { "code": "W9901", "message": "リクエストパラメータが不正です（title）" } }
```

HTTP 400。`createRoute` の `request.body` スキーマで弾かれ、`honoOpenApiFactory` の `defaultHook` が `W9901` に変換しています。`invalidParameters` に不正項目名（`title`）が入る点に注目。

### 2段目: ドメインルール違反（validate → ZodError → globalErrorHandler）

更新で `title` も `author` も送らない（domain の `refine`「どちらか必須」違反）:

```sh
curl -s -X PUT http://localhost:3000/books/$ID \
  -H 'Content-Type: application/json' -d '{}' | jq
```

`validateUpdateBook()` 内の `validatedUpdateBookSchema.parse()` が `ZodError` を投げ、`globalErrorHandler` が `W9901` に変換します。

### 業務エラー: 存在しない / 重複

```sh
# 存在しない id → W0101（404）
curl -s http://localhost:3000/books/00000000-0000-0000-0000-000000000000 | jq

# 同じタイトルで2回作成 → 2回目は W0103（400）
curl -s -X POST http://localhost:3000/books -H 'Content-Type: application/json' -d '{"title":"重複本","author":"著者"}' >/dev/null
curl -s -X POST http://localhost:3000/books -H 'Content-Type: application/json' -d '{"title":"重複本","author":"著者"}' | jq
```

`W0101`/`W0103` は application 層の usecase（`nullOrUndefinedCheck`/`checkBookTitleExists`）が返したコードで、`setResponse` が日本語メッセージと HTTP ステータスに対応付けています。

## 5. ここまでで確認できたこと

1リクエストで全層を通過していることを、ログ（実行 SQL）とレスポンスで確認できます。

```
POST /books
  → bookRouter: c.req.valid('json') で検証 → validateCreateBook で domain 型へ
  → bookCreateUsecase: createBook() でコマンド生成 → checkBookTitleExists → repository.save()
  → bookRepository: executeTransaction 内で INSERT、returning で成否判定
  → setResponse: { apiStatus, data } に整形して返却
```

| 確認項目 | 対応する層・仕組み |
| --- | --- |
| Swagger に6本表示される | interface: `createRoute` の宣言から自動生成 |
| 不正リクエストが 400 + W9901 | infrastructure: `defaultHook` / `globalErrorHandler` |
| 見つからない 404 + W0101 | application: `nullOrUndefinedCheck` |
| 重複 400 + W0103 | application: `checkBookTitleExists` |
| 保存/取得が成功する | infrastructure: Drizzle 実装 + 境界 parse |
