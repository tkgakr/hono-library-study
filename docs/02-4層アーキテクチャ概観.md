# 02. 4層アーキテクチャの概観

本教材は、ドメイン駆動／クリーンアーキテクチャ風の **4層構成**を採用します。この章では「どの層が何を担当し、リクエストがどう流れるか」を頭に入れます。コードはまだ書きません。地図を持ってから歩き始めるための章です。

## 4つの層

図書館プロジェクトのディレクトリ構成はこうなります。

```
src/api/
├─ domain/          ← ビジネスのルールと型。何にも依存しない一番内側
│  ├─ model/        … zod スキーマ・型・ドメイン関数（create/update…）
│  ├─ repository/   … リポジトリの「インターフェース」(I...Repository)
│  └─ service/      … logger / date など抽象インターフェース
├─ application/     ← ユースケース（処理の段取り）
│  ├─ usecase/      … 1処理1ファイル（create / getList / update …）
│  └─ service/      … neverthrow 用の共通ユーティリティ・チェック
├─ infrastructure/  ← 外界とのつなぎ込み（DB・ログ・設定）
│  ├─ database/     … Drizzle テーブル定義・リポジトリ「実装」
│  ├─ middleware/   … エラーハンドリング・レスポンス整形・OpenAPIHono生成
│  └─ logger/ config/ date/ … サービスの具体実装
└─ interface/       ← 外向きの入出力（HTTP / OpenAPI）
   ├─ model/        … リクエスト/レスポンスの zod-openapi スキーマと変換関数
   ├─ router/       … Hono ルート定義（createRoute）
   └─ routerIndex.ts … ルート集約
```

`src/test/` は本体と同じレイヤー構成でテストを置きます（最終章で扱います）。

## 依存の向き（ここが最重要）

層は **内側にしか依存できません**。矢印は「import してよい向き」です。

```
interface ─┐
           ├─→ application ─→ domain ←─┐
infrastructure ─────────────────────┘
```

- `domain` は誰も import しない（一番内側・純粋）
- `application` は `domain` だけに依存する
- `interface` と `infrastructure` は外側。`application`/`domain` に依存してよい
- **上位の層のファイルを下位が import してはいけない**（例: `domain` が `application` を import するのは禁止）

これを守ると、ビジネスルール（domain）が「DB が Postgres であること」「HTTP で来ること」を一切知らずに済みます。だから DB やフレームワークを差し替えてもドメインは無傷です。

### 「インターフェースは内側、実装は外側」

DB アクセスのような外界依存は、**抽象（インターフェース）を内側に、具体（実装）を外側に**置くことで依存の向きを保ちます。これが依存性逆転です。

| 抽象（domain 側） | 実装（infrastructure 側） |
| --- | --- |
| `IBookRepository`（repository/） | `bookRepository`（database/repository/） |
| `ILogger`（service/logger/） | `textLogger`（logger/） |
| `IDateService`（service/date/） | `dateService`（date/） |

application 層は `IBookRepository` という「契約」だけを見て処理を書き、実物の DB 実装はルーターで注入します。テストではこの契約をモックに差し替えるだけで、DB なしにユースケースを検証できます。

## リクエストが流れる順番

`GET /books/{id}` を例に、1リクエストの一生を追います（図書館プロジェクト版）。

```
HTTP リクエスト
   │
   ▼
[interface/router]  createRoute で宣言したスキーマで自動バリデーション
   │   c.req.valid('param') で検証済みの値を取り出す
   │   interface/model の validate...() で domain の型へ変換
   ▼
[application/usecase]  repository(契約) と logger を受け取り処理を段取り
   │   neverthrow の Result で「成功 or エラーコード」を返す
   ▼
[domain]  ビジネスルール（存在チェック・状態チェック）
   │
   ▼
[infrastructure/repository]  Drizzle で実際に DB アクセス
   │   取得結果を domain スキーマで parse して境界で型を保証
   ▼
[application] へ戻り、Result を組み立て
   │
   ▼
[interface/router]  result.isOk() で分岐し setResponse(c, ...) で統一形式に整形
   │
   ▼
HTTP レスポンス（{ apiStatus: {code, message}, data } の固定エンベロープ）
```

ポイントは2つです。

1. **バリデーションが2段ある**
   - 1段目: `createRoute` に宣言したスキーマ → `OpenAPIHono` の `defaultHook` が自動で弾く（形式エラー → `W9901`）
   - 2段目: `validate...()` 内の `domain スキーマ.parse()` → 失敗時の `ZodError` を `globalErrorHandler` が拾う

2. **エラーは例外で投げず、`Result`（neverthrow）で返す**
   application 層は `try/catch` をほぼ書かず、「成功値 or `OperationResult`(エラーコード)」を返します。HTTP ステータスや日本語メッセージへの変換は interface 層（`setResponse`）の仕事です。

## この章のまとめ

- 層は **domain ← application ← (interface / infrastructure)** の順に内→外
- 内側ほど純粋（domain は何も知らない）、外側ほど具体（DB・HTTP）
- 抽象（`I...`）を内側、実装を外側に置いて依存を逆転させる
- リクエストは interface → application → domain → infrastructure と流れ、戻りで `Result` をレスポンスへ整形する

次章から、この地図の **一番外側の足場（DB 基盤）** を作り、`book` を題材に domain → infrastructure → application → interface と内側から組み上げていきます。
