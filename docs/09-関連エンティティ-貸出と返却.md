# 09. 関連エンティティ — 利用者と貸出（join・日付・ステータス）

book の縦1本が通ったので、次は**関連を持つエンティティ**に進みます。ここで新しく学ぶのは、book ではまだ出てこなかった3つのパターンです。

1. **日付の変換**: HTTP は ISO 文字列で日付を受け取り、`Date` に変換して下位層へ渡す
2. **ステータスの算出**: 保存された日付から「貸出中／延滞／返却済」を導出する
3. **join 集約レスポンス**: 貸出に紐づく書名・利用者名をまとめて返す

題材は **利用者(member)** と **貸出(loan)** です。member は book とほぼ同型なので差分だけ示し、loan を丁寧に扱います。

```
loan（貸出）
├─ bookId    … どの蔵書を
├─ memberId  … 誰が
├─ loanedOn  … いつ借りて（Date）
├─ dueOn     … いつまでに返す（Date）
└─ returnedOn… いつ返したか（Date | null）← null なら未返却
```

操作は **貸出(create) / 返却(return) / 一覧取得(getList、join あり)** の3つにします。

## 1. 利用者 member（book とほぼ同型・差分のみ）

member は `{ id, name, email, isActive }`。book の `title/author` を `name/email` に置き換えるだけで、4層すべて book と同じ構造です。結果コードは `02` 帯（`W0201`/`E0201`…）を割り当てます。

```ts
// src/api/domain/model/member/member.ts （抜粋・book と同型）
export const getMemberSchema = z
  .object({
    id: z.uuid(),
    name: notBlankStringSchema,
    email: z.email(),
    isActive: z.boolean(),
  })
  .brand<'GetMember'>()
export type GetMember = z.infer<typeof getMemberSchema>
// validatedCreateMemberSchema / createMember / ... は book と同じパターンなので省略
```

> member の各層は04〜07章の book を雛形にそのまま写経できます。ここでは loan が member を参照するため「member は存在する」前提で進めます。

## 2. domain: loan モデル `domain/model/loan/loan.ts`

新しいのは **日付フィールドと、日付をまたいだ `refine`** です。

```ts
// src/api/domain/model/loan/loan.ts
import z from 'zod'

export const loanSaveOperations = {
  CREATE: 'create',
  RETURN: 'return',
} as const

// ステータスは保存された日付から算出する派生値（DBには持たない）
export const loanStatuses = {
  ON_LOAN: 'onLoan', // 貸出中
  OVERDUE: 'overdue', // 延滞
  RETURNED: 'returned', // 返却済
} as const
export type LoanStatus = (typeof loanStatuses)[keyof typeof loanStatuses]

// DB から取得した1件
export const getLoanSchema = z
  .object({
    id: z.uuid(),
    bookId: z.uuid(),
    memberId: z.uuid(),
    loanedOn: z.date(),
    dueOn: z.date(),
    returnedOn: z.date().nullable(),
  })
  .brand<'GetLoan'>()
export type GetLoan = z.infer<typeof getLoanSchema>

// (1) 貸出：返却期限は貸出日以降でなければならない（日付をまたぐ refine）
export const validatedCreateLoanSchema = z
  .object({
    bookId: z.uuid(),
    memberId: z.uuid(),
    loanedOn: z.date(),
    dueOn: z.date(),
  })
  .refine((data) => data.dueOn.getTime() >= data.loanedOn.getTime(), {
    message: '返却期限は貸出日以降にしてください',
    path: ['dueOn'],
  })
  .brand<'ValidatedCreateLoan'>()
export type ValidatedCreateLoan = z.infer<typeof validatedCreateLoanSchema>

export const createdLoanSchema = z
  .object({
    operation: z.literal(loanSaveOperations.CREATE),
    id: z.uuid(),
    bookId: z.uuid(),
    memberId: z.uuid(),
    loanedOn: z.date(),
    dueOn: z.date(),
  })
  .brand<'CreatedLoan'>()
export type CreatedLoan = z.infer<typeof createdLoanSchema>

export const createLoan = (entity: ValidatedCreateLoan): CreatedLoan =>
  createdLoanSchema.parse({ operation: loanSaveOperations.CREATE, id: crypto.randomUUID(), ...entity })

// (2) 返却：返却日を記録する
export const returnedLoanSchema = z
  .object({
    operation: z.literal(loanSaveOperations.RETURN),
    id: z.uuid(),
    returnedOn: z.date(),
  })
  .brand<'ReturnedLoan'>()
export type ReturnedLoan = z.infer<typeof returnedLoanSchema>

export const returnLoan = (id: string, returnedOn: Date): ReturnedLoan =>
  returnedLoanSchema.parse({ operation: loanSaveOperations.RETURN, id, returnedOn })

export type SaveLoan = CreatedLoan | ReturnedLoan

// (3) ステータス算出：保存値（日付）からドメインのルールで導く純粋関数
export const resolveLoanStatus = (loan: GetLoan, today: Date): LoanStatus => {
  if (loan.returnedOn !== null) return loanStatuses.RETURNED
  return loan.dueOn.getTime() < today.getTime() ? loanStatuses.OVERDUE : loanStatuses.ON_LOAN
}
```

ポイント:
- **日付は domain では `Date` で扱う**（`z.date()`）。文字列ではない。文字列⇄Date の変換は interface 層の責務（後述）。
- **`refine` で日付の前後関係を検証**（`dueOn >= loanedOn`）。book の「どちらか必須」より一歩進んだクロスフィールド検証。
- **`resolveLoanStatus` は純粋関数**。DB にステータス列は持たず、`returnedOn` と `dueOn` から毎回導出する。`today` を引数で受けて決定的にする（テストしやすい）。

## 3. domain: 集約レスポンス型とリポジトリ契約

一覧では「貸出 + 書名 + 利用者名 + ステータス」をまとめて返したい。これを**集約レスポンス型**として domain に定義します。

```ts
// src/api/domain/model/loan/loanListItem.ts
import { z } from 'zod'
import { loanStatuses } from '@domain/model/loan/loan'

// 一覧の1行（join 済みの集約ビュー）
export const loanListItemSchema = z
  .object({
    id: z.uuid(),
    bookTitle: z.string(),
    memberName: z.string(),
    loanedOn: z.date(),
    dueOn: z.date(),
    returnedOn: z.date().nullable(),
    status: z.enum([loanStatuses.ON_LOAN, loanStatuses.OVERDUE, loanStatuses.RETURNED]),
  })
  .brand<'LoanListItem'>()
export type LoanListItem = z.infer<typeof loanListItemSchema>
```

```ts
// src/api/domain/repository/loan/loanRepository.ts
import type { GetLoan, SaveLoan } from '@domain/model/loan/loan'
import type { LoanListItem } from '@domain/model/loan/loanListItem'
import type { EntityData, ListData } from '@domain/model/generic/repositoryData'

export interface ILoanRepository {
  fetchListWithRelations: () => Promise<ListData<LoanListItem>> // join 集約

  save: (command: SaveLoan) => Promise<boolean>

```

> 一覧（join 済みの表示用ビュー）と詳細（生の `GetLoan`）で返す型が異なる点に注目。表示集約は `LoanListItem`、業務ロジック用は `GetLoan` と使い分けます。

## 4. infrastructure: loan テーブルと join 取得

テーブルは book/member への外部キーを持ちます（学習用なので参照整合性は緩め）。一覧取得で **book/member を join** し、ステータスを算出して `LoanListItem` に組み立てます。

```ts
// src/api/infrastructure/database/model/loan/loan.ts （抜粋）
import { defaultTimestamps, primaryId } from '@infrastructure/database/model/generic/commonColumns'
import { date, pgTable, uuid } from 'drizzle-orm/pg-core'

const columns = {
  ...defaultTimestamps,
  id: primaryId,
  bookId: uuid().notNull(),
  memberId: uuid().notNull(),
  loanedOn: date({ mode: 'date' }).notNull(),
  dueOn: date({ mode: 'date' }).notNull(),
  returnedOn: date({ mode: 'date' }), // nullable
} as const

export const loanTableConfig = { name: 'loan', columns }
const loanTable = pgTable(loanTableConfig.name, columns)
export default loanTable
// insert/return の parse 関数（createInsertSchema / createUpdateSchema）は book と同じパターンで作る
```

```ts
// src/api/infrastructure/database/repository/loanRepository.ts （fetchListWithRelations 抜粋）
import { getDbInstance } from '@infrastructure/database/dbAccess'
import loanTable from '@infrastructure/database/model/loan/loan'
import bookTable from '@infrastructure/database/model/book/book'
import memberTable from '@infrastructure/database/model/member/member'
import { loanListItemSchema } from '@domain/model/loan/loanListItem'
import { resolveLoanStatus } from '@domain/model/loan/loan'
import { eq } from 'drizzle-orm'

const fetchListWithRelations = async () => {
  const db = getDbInstance()
  // join して必要なカラムだけ取得（DTO 投影）
  const rows = await db
    .select({
      id: loanTable.id,
      bookTitle: bookTable.title,
      memberName: memberTable.name,
      loanedOn: loanTable.loanedOn,
      dueOn: loanTable.dueOn,
      returnedOn: loanTable.returnedOn,
    })
    .from(loanTable)
    .innerJoin(bookTable, eq(loanTable.bookId, bookTable.id))
    .innerJoin(memberTable, eq(loanTable.memberId, memberTable.id))

  const today = new Date()
  // 取得した行を集約レスポンス型に組み立てる（ステータスはドメイン関数で算出）
  const value = rows.map((row) =>
    loanListItemSchema.parse({
      ...row,
      status: resolveLoanStatus(
        { id: row.id, bookId: '', memberId: '', loanedOn: row.loanedOn, dueOn: row.dueOn, returnedOn: row.returnedOn } as never,
        today,
      ),
    }),
  )
  return { value, total: value.length }
}
```

ポイント:
- **join した DTO を、いったんドメインの考え方で組み立て直してから返す**（join を伴う取得は、取得 DTO をドメイン型へ変換した上で集約レスポンスを組み立てるのが定石）。
- ステータスは取得時に `resolveLoanStatus` で算出して埋める。DB には持たない。
- 境界で `loanListItemSchema.parse()`。

## 5. application: 貸出ユースケース（日付は受け取るだけ）

usecase は book と同じ neverthrow パターン。日付は既に `Date` に変換されて渡ってくるので、ビジネスルール（例：同じ蔵書が貸出中でないか）に集中します。

```ts
// src/api/application/usecase/loan/loanCreateUsecase.ts （抜粋）
export const loanCreateUsecase = async (
  repository: ILoanRepository,
  logger: ILogger,
  validatedEntity: ValidatedCreateLoan,
): Promise<Result<boolean, OperationResult>> => {
  return ok(createLoan(validatedEntity))
    .asyncAndThen((created) => ResultAsync.fromPromise(repository.save(created), promiseErrorReturn(logger, ResultCodes.LOAN_SAVE_FAILED)))
    .andThen(falsyValueCheck(ResultCodes.LOAN_SAVE_FAILED))
}
```

> 「その蔵書が既に貸出中なら貸せない」等の追加ルールは、book の `checkBookTitleExists` と同じ要領で `loanCheckService` を作って `.andThrough` で差し込みます（演習）。

## 6. interface: ISO 文字列 ⇄ Date の変換（最重要ポイント）

ここが book との一番の違いです。**HTTP では日付を ISO 文字列で受け取り、`validate*()` で `Date` に変換**してから domain へ渡します。

```ts
// src/api/interface/model/loan.ts （作成部分の抜粋）
import { z } from '@hono/zod-openapi'
import { validatedCreateLoanSchema } from '@domain/model/loan/loan'
import type { ValidatedCreateLoan } from '@domain/model/loan/loan'

// 生の HTTP：日付は ISO 文字列で受ける
export const unValidatedCreateLoanSchema = z
  .object({
    bookId: z.uuid().openapi({ example: '...' }),
    memberId: z.uuid().openapi({ example: '...' }),
    loanedOn: z.iso.date().openapi({ description: '貸出日(YYYY-MM-DD)', example: '2026-06-22' }),
    dueOn: z.iso.date().openapi({ description: '返却期限(YYYY-MM-DD)', example: '2026-07-06' }),
  })
  .brand<'UnValidatedCreateLoan'>()
export type UnValidatedCreateLoan = z.infer<typeof unValidatedCreateLoanSchema>

// 継ぎ目：文字列 → Date に変換してから domain の parse へ
export const validateCreateLoan = (request: UnValidatedCreateLoan): ValidatedCreateLoan =>
  validatedCreateLoanSchema.parse({
    bookId: request.bookId,
    memberId: request.memberId,
    loanedOn: new Date(request.loanedOn),
    dueOn: new Date(request.dueOn),
  })
```

レスポンス側は逆に、`Date` を ISO 文字列に整形して返します（`LoanListItem` を JSON 化する `formatLoanListResponse` のような変換関数を model に置くのが定石）。

```ts
// 例：レスポンス整形（Date → ISO 文字列、status はそのまま）
export const formatLoanListItem = (item: LoanListItem) => ({
  ...item,
  loanedOn: item.loanedOn.toISOString().slice(0, 10),
  dueOn: item.dueOn.toISOString().slice(0, 10),
  returnedOn: item.returnedOn ? item.returnedOn.toISOString().slice(0, 10) : null,
})
```

ルーター（`createRoute` + `openapi`）と集約は book と同じ形なので省略します。`GET /loans` を叩くと、書名・利用者名・算出済みステータス付きの一覧が返ります。

## この章のまとめ

新しく学んだ3パターン:

| パターン | どの層で | 要点 |
| --- | --- | --- |
| 日付変換 | interface | HTTP は ISO 文字列 / domain は `Date`。`validate*()` で変換、レスポンスで逆変換 |
| ステータス算出 | domain | DB に持たず、`resolveLoanStatus` の純粋関数で日付から導出。`today` を引数化 |
| join 集約 | infrastructure | join した DTO をドメイン型へ組み立て、`LoanListItem` で境界 parse |
| クロスフィールド検証 | domain | `refine` で `dueOn >= loanedOn` |

book で学んだ4層の骨格は完全に再利用しつつ、関連・日付・派生値という現実的な複雑さを足せました。次章では、ここまで作った各層に**テスト**を書きます。
